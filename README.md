IMPLEMENTATION (RISC-V)

Instruction Set Architecture :

The processor supports the standard RV32I base instruction set: includes arithmetic and logical operations, load and store instructions, control flow instructions and system-level instructions
All instructions follow a fixed 32-bit format, simplifying instruction decoding and enabling efficient hardware implementation. In addition to the base instruction set, the RV32M extension introduces multiplication, division, and remainder operations. 
These operations are implemented using hardware multiplication units and iterative division algorithms to reduce hardware complexity while maintaining computational capability.

Multi-Cycle Execution:

The processor adopts a multi-cycle execution architecture in which instruction processing is divided into sequential stages rather than being completed in a single clock cycle. Typical execution stages include instruction fetch, instruction decode, execution or address computation, memory access when required, and register writeback. 
This staged execution allows reuse of hardware resources, reduces critical path delay, and improves timing performance, making the design suitable for FPGA-based embedded implementations. 
The multi-cycle approach also facilitates integration of complex operations such as division and custom hardware interactions without significantly increasing hardware complexity

Control Unit, Memory Interface:

Processor operation is controlled by a finite state machine that manages transitions between instruction fetch, execution, memory access, and wait states. This ensures proper synchronization with memory latency and multi-cycle arithmetic operations.  
The processor supports byte, half-word, and word memory accesses, with handshake signals used to safely manage memory readiness and prevent data hazards. A 32-entry general-purpose register file with dual read ports and a single write port provides efficient operand access, while register x0 remains permanently tied to zero as specified by the RISC-V architecture.
Overall, the processor design achieves efficient hardware utilization, extensibility through custom ISA integration, reliable multi-cycle execution, and suitability for embedded FPGA-based system-on-chip applications.

Custom Extended Instruction Set (Custom ISA Integration):

A significant feature of the processor is the integration of a custom instruction extension using reserved opcode space defined in the RISC-V architecture. This enables the addition of application-specific instructions that can activate dedicated hardware accelerators. 
When a custom instruction is detected during the decode stage, the processor generates control signals to initiate the specialized hardware operation. Parameters required for the operation are passed through general-purpose registers, and the processor may enter a wait state until the operation is completed. 
This approach allows efficient hardware acceleration of computationally intensive tasks, reduces processor workload, and enhances system performance while preserving ISA flexibility and compatibility.

System Configuration and Initialization:

The implementation begins with system configuration through the RISC-V processor, which provides the start control signal and image parameters as memory base addresses. Upon reset, all internal registers, line buffers, address counters, and pipeline stages are cleared to ensure deterministic operation. Once the start signal is asserted, the Sobel accelerator begins sequential image processing, streaming pixels directly from memory into the hardware pipeline.

Streaming Control and Execution Flow:

Instead of a complex centralized FSM, the design follows a streaming datapath architecture where control is derived from valid signals propagating through the pipeline. Pixel validity generated by the input address generator drives window formation, filtering, Sobel computation, and output write-back. This approach simplifies control logic while maintaining continuous throughput and synchronization between stages.

Sequential Memory Access and Address Generation:

An input address generator module computes pixel addresses using row and column counters in raster-scan order. Pixels are read sequentially from memory as a continuous grayscale stream. This linear addressing scheme avoids random memory access, enabling efficient pipelined processing and predictable memory bandwidth utilization.

Line Buffering and Window Formation:

To support the 3×3 convolution required for edge detection, the design uses three line buffers implemented as memory arrays. As each new pixel arrives, older rows are shifted vertically while horizontal shift registers track neighboring column pixels. This structure continuously produces a valid 3×3 pixel window without redundant memory reads, improving efficiency and throughput.

Sobel Gradient Computation Pipeline:

The Sobel computation module calculates horizontal (Gx) and vertical (Gy) gradients using shift-and-add arithmetic on the 3×3 pixel window. Absolute gradient components are summed to estimate edge magnitude, with saturation logic preventing overflow. The design is pipelined so that, once filled, one edge pixel is produced per clock cycle.

Gaussian Filtering:

Before Sobel gradient computation, Gaussian smoothing stage performs weighted averaging on the 3×3 window to reduce noise sensitivity. This filtering improves edge stability and accuracy, especially in noisy image environments, while maintaining pipeline compatibility.

Output Address Generation and Write-Back:

An output address generator computes write addresses based on delayed row and column indices aligned with pipeline latency. Valid Sobel outputs are written sequentially back to memory, excluding boundary pixels where a full convolution window is unavailable. This ensures spatial alignment between input and processed output images.

Pipeline Completion and Continuous Operation:

After the final pixel is processed, remaining pipeline data is flushed automatically through valid-signal propagation. Once all outputs are written, valid signals de-assert, indicating completion. The system then remains ready for the next start command without requiring extensive reinitialization..
